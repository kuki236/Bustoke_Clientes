## ⚠️ Notas y discrepancias detectadas

1. **TC-BB-031 (Claims detalle)**: el plan dice "400 Bad Request" pero en el schema real (`claim_schema.py:19`) `detalle: min_length=1`, NO 15. El frontend SÍ muestra el mensaje de "15 caracteres" en `claims.js:30-32` como validación defensiva del cliente, pero el backend rechaza con 422 (no 400) si `detalle=""` o con 201 si `detalle` tiene ≥ 1 carácter. **Recomendación**: actualizar el caso para que diga "422 si se bypasea el frontend" y "Frontend muestra error de 15 chars antes de enviar".

2. **TC-RB-002 (Pérdida de plata)**: el plan asume que el backend devuelve 409, pero no hay validación implementada en `booking_service.py` que distinga explícitamente entre "asiento vendido" vs "pago MP ya procesado". El `mp_payment_id` se persiste en `pagos.referencia_transaccion` para trazabilidad, pero NO hay un check que diga "este mp_payment_id ya fue usado". Esto es un gap real de implementación — **debería agregarse un test que verifique idempotencia de mp_payment_id**.

3. **TC-RB-003 (Holds zombie)**: el plan asume que hay un job de limpieza de holds expirados. **No lo hay** en el código actual — la única mitigación es `releaseHoldsBeacon` (que solo funciona si la pestaña emite el beacon antes de cerrar). El test pasaría en el caso del beacon pero fallaría en el caso de cierre abrupto (kill -9, crash del browser).

4. **TC-RB-005 (Doble embarque)**: ✅ verificado, el código usa `with_for_update()` correctamente.

5. **TC-WB-009 (corregido)**: el nombre del campo es `localPaymentId` (no `__mockPaymentId`). La fila de la tabla ya fue corregida para decir "cardFormData.localPaymentId" en vez de "__mockPaymentId".
